# Dietician Patient Mobile App

React Native (Expo) patient-facing app for the Dietician platform. Patients log in with credentials generated by the dietician, view their assigned diet plan, and receive scheduled notifications for meals, water intake, wake-up, and sleep reminders.

## Authentication Flow

1. Patient opens the app — `onAuthStateChanged` in `AppNavigator` fires.
2. If a Firebase session exists → `fetchUserProfile(email)` queries Firestore `users` collection by `userEmail` field → profile loaded into Zustand.
3. If no session → `LoginScreen` is shown.
4. Login accepts **email** (`xxx@dietapp.com`) or the shorter **userId** (`priya2026abc`). If no `@` is detected, `resolveEmailByUserId` does a Firestore lookup to resolve it to the full email before calling `signInWithEmailAndPassword`.
5. On sign-out → Firebase session cleared, Zustand `reset()`, all scheduled notifications cancelled.

## Notification System

Implemented using `expo-notifications`. All scheduling happens at login; notifications run entirely in the background — **tapping an action button never opens the app UI**.

### Packages

```
expo-notifications
expo-task-manager   (declared as peer, used for background task registration)
```

### Setup (called on every app boot in `AppNavigator`)

1. `setupAndroidChannel()` — creates Android notification channel `dietplan` (HIGH importance, indigo light colour)
2. `setupNotificationCategories()` — registers two action button categories:
   - `MEAL_COMPLETE` → button: `"✅ Mark Complete"`, `opensAppToForeground: false`
   - `WATER_DONE` → button: `"💧 I Drank It"`, `opensAppToForeground: false`
3. `requestPermissions()` — prompts iOS/Android for notification permission
4. `addNotificationResponseReceivedListener(handleNotificationResponse)` — handles action taps when app is in foreground or background
5. `Notifications.getLastNotificationResponseAsync()` — handles the case where app was killed, user tapped the action button, and the app boots fresh

### Scheduling (called after login once the active plan is loaded)

`scheduleAllNotifications(plan, userId)`:

| Type | Trigger | Count | Action button |
|---|---|---|---|
| Wake-up | Daily at `plan.wakeUpTime` | 1 | None |
| Bedtime | Daily at `plan.sleepTime` | 1 | None |
| Water | Daily at each `waterSchedule[].time` slot | up to 16 | `"💧 I Drank It"` |
| Meal | Date-specific, 20 min before each meal, next 7 days | up to 21 | `"✅ Mark Complete"` |

**Total scheduled ≤ 39** — well under the iOS 64-notification limit.

Meal notifications are re-scheduled on every login, rolling the 7-day window forward. Water and wake/sleep are daily-repeating so they never expire.

### Notification identifier encoding

All data needed to write the Firestore log is encoded in the notification identifier so the background handler never needs app state:

```
meal__YYYY-MM-DD__mealId__mealName__HH:MM__planId__dayIndex
water__HH-MM__amountMl__planId
wakeup    (static — no log written)
sleep     (static — no log written)
```

### Action handler (`handleNotificationResponse`)

1. Checks `actionIdentifier === 'complete' | 'done'` — ignores bare notification taps
2. Reads `userId` from AsyncStorage (stored at login, cleared on sign-out)
3. Parses the identifier → reconstructs `MealLog` or `WaterLog` payload
4. Calls `logMealCompletion` or `logWaterCompletion` → writes to Firestore
5. Calls `Notifications.dismissNotificationAsync` → removes notification from panel

### Platform behaviour

| Scenario | Behaviour |
|---|---|
| App in foreground | `addNotificationResponseReceivedListener` fires immediately |
| App in background | Listener fires silently — UI stays in background |
| App killed, action tapped | App boots silently; `getLastNotificationResponseAsync()` picks up the response |
| Android battery optimisation | Aggressive Doze mode may defer background wake — user may need to whitelist the app |

## Firestore Schema

### `users/{userId}` (existing)

```typescript
{
  id:               string    // Firestore document ID
  name:             string
  age:              number
  gender:           string
  phone?:           string
  weight:           number    // kg
  height:           number    // cm
  bmi:              number
  bmiCategory:      string
  bodyType:         string
  bodyFatPercent:   number | null
  muscleMass:       number | null
  boneMass:         number | null
  bodyWaterPercent: number | null
  visceralFat:      number | null
  bmr:              number | null
  metabolicAge:     number | null
  goal:             string
  preference:       string
  allergies:        string[]
  conditions:       string[]
  medications:      string
  notes:            string
  userId:           string    // login ID e.g. "priya2026abc"
  userEmail:        string    // e.g. "priya2026abc@dietapp.com"
  status:           "active" | "no-plan" | "inactive"
  createdAt:        string
  updatedAt:        string
}
```

### `users/{userId}/dietPlans/{planId}` (existing)

```typescript
{
  templateId:    string | null
  templateName:  string
  days: [{
    day:        number      // 1-based
    dayName:    string
    isOverride: boolean
    meals: [{
      id:       string      // random alphanumeric — used as mealId in logs
      name:     string      // "Breakfast"
      time:     string      // "HH:MM" 24h
      items:    [{ name, calories?, protein?, carbs?, fats? }]
      calories: number
      protein:  number
      carbs:    number
      fats:     number
      notes:    string
    }]
  }]
  wakeUpTime:    string     // "HH:MM"
  sleepTime:     string     // "HH:MM"
  waterIntakeMl: number     // ml e.g. 2000
  waterSchedule: [{ time: string, amountMl: number }]
  tips:          string
  assignedAt:    string     // ISO 8601
  assignedBy:    string
  status:        string     // "active"
}
```

### `users/{userId}/mealLogs/{logId}` ← **NEW**

Document ID: `{YYYY-MM-DD}__{mealId}` — idempotent, no duplicates on reschedule.

```typescript
{
  planId:        string    // dietPlans document ID
  date:          string    // "YYYY-MM-DD"
  dayIndex:      number    // 0-based index into plan.days
  mealId:        string    // meal.id from the plan
  mealName:      string    // "Breakfast"
  scheduledTime: string    // "HH:MM" original meal time
  completed:     boolean   // true when patient taps "Mark Complete"
  completedAt:   string | null  // ISO 8601 timestamp or null
}
```

### `users/{userId}/waterLogs/{logId}` ← **NEW**

Document ID: `{YYYY-MM-DD}__{HH-MM}` — idempotent.

```typescript
{
  planId:        string
  date:          string    // "YYYY-MM-DD"
  scheduledTime: string    // "HH:MM"
  amountMl:      number    // ml scheduled for this slot
  completed:     boolean   // true when patient taps "I Drank It"
  completedAt:   string | null
}
```

## How the Dashboard Reads This Data

The dashboard (React + TanStack Query) can query `mealLogs` and `waterLogs` from the patient profile page to build an adherence report.

### Firestore queries

```typescript
import { collection, query, where, orderBy, getDocs } from 'firebase/firestore';

// All meal logs for a patient in a date range
const mealLogsQuery = query(
  collection(db, 'users', patientId, 'mealLogs'),
  where('date', '>=', '2026-03-01'),
  where('date', '<=', '2026-03-31'),
  orderBy('date', 'asc'),
);

// All water logs for a patient in a date range
const waterLogsQuery = query(
  collection(db, 'users', patientId, 'waterLogs'),
  where('date', '>=', '2026-03-01'),
  where('date', '<=', '2026-03-31'),
  orderBy('date', 'asc'),
);

const [mealSnap, waterSnap] = await Promise.all([
  getDocs(mealLogsQuery),
  getDocs(waterLogsQuery),
]);

const mealLogs  = mealSnap.docs.map(d => d.data());
const waterLogs = waterSnap.docs.map(d => d.data());
```

> **Firestore index required:** Compound index on `mealLogs` for `(date ASC)` and on `waterLogs` for `(date ASC)`. Firestore will prompt you to create these the first time the query runs in the Firebase console.

### Adherence calculations the dashboard can derive

```typescript
// Meal adherence % for a given day
const mealsForDay   = mealLogs.filter(l => l.date === '2026-03-18');
const completedMeals = mealsForDay.filter(l => l.completed).length;
const mealAdherence  = (completedMeals / mealsForDay.length) * 100;

// Water adherence % for a given day
const waterForDay    = waterLogs.filter(l => l.date === '2026-03-18');
const completedSlots = waterForDay.filter(l => l.completed).length;
const waterAdherence = (completedSlots / waterForDay.length) * 100;

// Total water consumed on a day (ml)
const waterConsumed = waterForDay
  .filter(l => l.completed)
  .reduce((sum, l) => sum + l.amountMl, 0);

// Weekly meal adherence trend
const byDate = mealLogs.reduce((acc, log) => {
  acc[log.date] = acc[log.date] ?? { total: 0, completed: 0 };
  acc[log.date].total++;
  if (log.completed) acc[log.date].completed++;
  return acc;
}, {} as Record<string, { total: number; completed: number }>);
```

### What to show in the UserReports page

| Widget | Data source |
|---|---|
| Meal adherence % per day (bar/line chart) | `mealLogs` grouped by `date`, `completed / total` |
| Water intake per day vs target (bar chart) | `waterLogs` sum of `amountMl` where `completed = true` vs `plan.waterIntakeMl` |
| Missed meals table | `mealLogs` where `completed = false`, grouped by date |
| Streak — consecutive days with 100% meal adherence | Computed from daily adherence map |
| Per-meal breakdown (Breakfast / Lunch / Dinner) | `mealLogs` filtered by `mealName` |
| Completion timestamp distribution | `completedAt` times → histogram of when patient eats |

## Monorepo React Version Fix

The dashboard uses React 19; the mobile app uses React 18. A custom Metro resolver in `metro.config.js` forces all `react` and `react-native` imports to resolve from `mobile/node_modules` to prevent hook call errors when Metro traverses the monorepo.

## Setup & Running

```bash
cd mobile
npm install

# Start dev server (Expo Go — UI only, no notifications)
npx expo start

# Run on Android device/emulator (full native build — required for notifications)
npx expo run:android

# Run on iOS simulator (full native build)
npx expo run:ios

# Production APK via EAS
npx eas build --platform android --profile preview
```
